Short Paragraph

  Our group assumed that some use cases were done by different systems so therefore they did not fall into the ChocAn system itself. First we produced a glossary and defined the actors and basic elements of the project. Some terms were defined with more detail in order to make the descriptions simple and not repeat what was already defined. Then we figured out the use cases by going line by line through the ChocAn project description. We determined the actors first then how each actor interacted with the system and with other actors. We made the assumption that the member itself does not interact with the system, instead the provider is the one who interacts with the system for the member. So the provider is the one that initiates the system after they provide services to the member. For the use case diagram we broke it down in the most general sense we could and then expanded on each of those based on if there were any more specifications for certain use cases. We decided which situations we should include on the diagram and which ones should be in the descriptions. We all brainstormed the use case diagram together on a white board to then determine the final diagram. To write the descriptions, we split them up evenly among group members. Each group member then read through the project description while writing the descriptions in order to provide as much detail as possible. One group member wrote the html code for the Report.html file and their individual file RyanKutellaTeam11Report.html. The other group members copied and pasted this outline for their own individual files and included their own use case diagrams. After all of the tasks were finished, the submitter, Ali Burkemper, input all of the tasks into the Report.html outline to finish the report file.

Glossary

Key Terms Brief Descriptions
ChocAn Organization dedicated to helping people addicted to chocolate
Plastic Member Card Includes members name and 9 digit member number on a magnetic strip on the back of the card.
ChocAn Computer Terminal Provider uses to access member’s current membership status (valid, invalid, suspended)
Provider Directory Provider uses to look up the 6 digit service code corresponding to the service that was provided to the member.
Software Product Writes the following to a disk: date/time (MM/DD/YYYY HH:MM:SS), date of service (MM/DD/YYYY), provider number (9 digits), member number (9 digits), service code (6 digits), and comments (100 characters).
Report Can be accessed at any time from request of a ChocAn manager, or, on Fridays at midnight reports are generated for the current week's services provided. A report is also sent to any members who consulted a ChocAn provider and includes member name, number, street address, city, state, zip code, and for each service provided includes: date of service, provider name, and service name. Providers also receive a report of services he or she provided to ChocAn members that week, with a total number of consultations and fees at the end.
Electronic Funds Transfer (EFT) A record written to a disk that makes sure the provider's bank account receives the correct amount.
Summary Report Report given to ChocAn managers and details every provider to be paid that week, the number of consultations each had, and his or her total fee for that week.
Acme Accounting Services Third party organization responsible for financial procedures, updating each evening at 9pm.
Provider The health care professional directly dealing with a member to treat their problems
ChocAn Manager ChocAn Manager's have the power to run reports ahead of the automated system by the ChocAn data center.
ChocAn Data Center The data center holds all data for members and communicates with terminals to provide member status, generate weekly reports, and can be used to add or delete members.
ChocAn Operator ChocAn operator's have the power to add, delete, and update new members.
Service code Retrieved by provider to verify to the ChocAn data centers the work being done to a member. Included on reports, retrieved from provider directory.

Use case: Verify member
Context:  The health care provider intends to verify the member’s number
Actors:   Provider
Main Success Scenario:  
  1. When the provider terminal detects a card, it dials the ChocAn data center.
   1.1. The ChocAn data center checks the member number in the system.
  2. The word “Validated” appears on a one-line display. 
Extensions:  
    1a. Invalid number 
      1a.1. The system displays “Invalid number”
    1b. Member account has been suspended because fees are owed
      1b.1. The system displays “Account suspended”

Use case: Run main accounting procedure
Context:  The main accounting procedure is run after notified by the timer and generates reports based in the week's provided services.
Actors:  Timer
Main Success Scenario: 
  1. At midnight on Friday the timer notifies the system.
  2. The system reads the file of the services provided from that week.
  3. Reports and summary reports are generated and printed.
  4. Reports are sent out to the members who received services that week, providers who provided services that week, and the manager.
Extensions:  
  No extensions.


Use case: Request Reports extends Run main accounting procedure
Context:  The manager can request reports that are a part of the main accounting procedure at any time during the week.
Actors:  Manager

  2. The system reads the file of the services that have been provided that week up to the current date.
  3. The reports are generated.
  4. The system sends the requested reports to the manager.
  5. Manager goes back to menu.

    5a. Manager logs out of terminal.

Use case: Add/bill for services
Context:  The provider uses the provider terminal to bill ChocAn after providing healthcare services to a member in order to receive payment.
Actors:  Provider
Main Success Scenario: 
  1. Provider passes the member’s card through the provider terminal
  1.1. The provider terminal dials the ChocAn data center
  1.2. ChocAn data center checks if the member number is in the system
  2. “Validated” appears on the provider terminal
  3. Provider inputs service info
  4. Software product writes a record to disk
   4.1. Software product includes the current date and time
   4.2. Software product includes the date the service was provided
   4.3. Software product includes the provider number
   4.4. Software product includes the member number
   4.5. Software product includes the service code of the service provided
   4.6. Software product includes provider comments
Extensions:  
    1a. Provider types in the member number into the provider terminal
    2a. The number is invalid
      2a.1. “Invalid number” displays on the terminal
    2b. Member account has been suspended because fees are owed
      2b.1. “Member suspended” displays on the terminal
      2b.2. Provider cancels billing process
    4a. There are no provider comments for the software product to include in the record


Use case: Input service info
Context:  Provider enters the date and service code for the corresponding service they provided into the provider terminal in order to bill ChocAn the correct amount.
Actors:  Provider
Main Success Scenario: 
  1. Provider inputs into the terminal the date the service was provided in the form MM-DD-YYYY
  2. Provider uses Provider Directory to look up the six-digit service code corresponding to the service they just provided
  3. Provider inputs service code into provider terminal
  4. Software product displays the name of the service corresponding to the input code on the provider terminal
   4.1. Software product asks provider to verify the name of the service
   4.2. Provider verifies service name
  5. Provider enters comments about the service provided
  6. Software product displays fee
Extensions:  
    2a. Provider knows service code from memory
    3a. Provider entered nonexistent code
      3a.1. An error message is displayed on the provider terminal
      3a.2. Provider inputs correct service code from the Provider Directory
    4a. The name of the service displayed is not the one that was provided
      4a.1. Provider does not verify service name
      4a.2. Provider inputs correct service code from the Provider Directory
    5a. Provider does not enter comments about the service


Use case: Display fee
Context:  Software product looks up and displays fee that the provider will be paid for providing a service.
Actors:  Provider
Main Success Scenario: 
  1. Software product looks up fee to be paid for the service provided based on the input service code
  2. Service fee is displayed on the providers terminal
  3. Provider completes verification form for service provided
   3.1. Provider enters current date and time on form in the provider terminal
   3.2. Provider enters the date the service was provided on form
   3.3. Provider enters the member name and number that the service was provided to
   3.4. Provider enters the service code corresponding to the service provide
   3.5. Provider enters the fee displayed on the terminal
  4. Provider goes back to menu.
Extensions:  
    4a. Provider logs out of terminal.
